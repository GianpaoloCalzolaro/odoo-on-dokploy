# Review di aderenza: `odoo-on-dokploy` vs `progetto_repository_template.md`

**Data:** 2026-07-17
**Revisione analizzata:** `b54d548` (branch `main`, allineato a `origin/main`)
**Ambito:** valutazione dell'aderenza del repository alle specifiche del brief e individuazione delle criticità strutturali, di sicurezza, di automazione e di configurazione da correggere immediatamente. Nessuna modifica al codice è stata applicata.

---

## 1. Executive summary

Il repository è oggi un **template di deploy minimale ma funzionante**: builda un'immagine custom basata su `odoo:18.0`, la pubblica su GHCR via GitHub Actions e la esegue su Dokploy con Docker Compose, PostgreSQL, volumi persistenti nominati e routing Traefik/HTTPS. La parte "infrastruttura di primo avvio" del brief è coperta in modo ragionevole.

Rispetto al brief complessivo, però, **l'aderenza è parziale e insufficiente per l'uso previsto** (riferimento tecnico permanente di istanze di produzione gestite con Coding Agent). Le distanze principali sono di quattro tipi:

1. **Sicurezza — una criticità bloccante attiva**: un **API key Dokploy reale, con l'IP del server**, è committato e pushato in `.claude/settings.json`. Oltre a essere un segreto di produzione versionato (vietato dal brief), dà a qualunque Coding Agent che opera sul repo accesso diretto al deploy di produzione, violando la separazione di responsabilità che è il cardine del brief.
2. **Tracciabilità e rollback assenti**: tutto gira su tag mobili (`:latest`, `odoo:18.0` senza digest, `pandas` senza versione, moduli OCA copiati a mano senza riferimento all'origine). Non è possibile ricostruire ciò che era in produzione in un dato momento né fare rollback — due requisiti espliciti del brief.
3. **Automazioni e flusso di collaborazione assenti**: la CI esegue solo build+push dell'immagine di produzione a ogni push su `main`, senza alcun controllo (lint, manifest, installazione moduli, test, avvio Odoo) e senza gate di pull request.
4. **Supporto al Coding Agent assente**: nessuna documentazione agent-oriented, nessun ambiente temporaneo di sviluppo/test, nessun comando di test definito.

**Giudizio sintetico per area** (dettaglio nella matrice, §2):

| Area del brief | Giudizio |
|---|---|
| Infrastruttura di produzione (compose, immagine, Traefik) | ⚠️ Parziale, buona base |
| Persistenza database e filestore | ⚠️ Parziale |
| Sicurezza e gestione dei segreti | ❌ Non conforme (1 bloccante) |
| Gestione dipendenze Python | ❌ Non conforme |
| Moduli esterni / standard repository esterni | ❌ Assente |
| Versionamento controllato / rollback | ❌ Assente |

| Supporto Coding Agent + ambiente temporaneo | ❌ Assente |
| Backup, ripristino, osservabilità | ⚠️ Parziale e in parte errato |
| Documentazione operativa | ⚠️ Parziale |

## 3. Criticità da correggere immediatamente

Ordinate per severità. Ogni voce indica file:riga, requisito violato, impatto e correzione raccomandata.

### 🔴 C-1 — API key Dokploy di produzione committata e pushata

- **Dove:** `.claude/settings.json:6-7` (commit `b54d548`, presente su `origin/main`; 2 occorrenze della chiave nella history del file)
- **Cosa:** la configurazione permessi Claude Code contiene in chiaro `DOKPLOY_API_KEY=mcpz…` e `DOKPLOY_URL=http://76.13.133.214:3000`.
- **Requisiti violati:** «Non devono essere versionati: … credenziali Dokploy, chiavi API»; «Il Coding Agent non deve avere accesso ai segreti di produzione»; «Il Coding Agent non può eseguire il deploy tramite Dokploy». Essendo questo un **template**, ogni repository derivato erediterebbe la chiave.
- **Impatto:** chiunque abbia accesso al repo (inclusi tutti i coding agent che vi operano, e chiunque se il repo è o diventa pubblico) può amministrare il server Dokploy: deploy, cancellazione progetti, lettura variabili d'ambiente di produzione (quindi anche password DB e admin Odoo di tutte le istanze). L'endpoint è inoltre in HTTP non cifrato.
- **Correzione immediata:**
  1. **Revocare/rigenerare subito la chiave API su Dokploy** (la rimozione dal repo non basta: è nella history remota).
  2. Rimuovere `.claude/settings.json` dal tracking o ripulirlo dai segreti; aggiungere `.claude/settings.local.json` a `.gitignore` e spostare lì ogni configurazione con credenziali.
  3. Valutare la riscrittura della history (`git filter-repo`) o considerare la chiave definitivamente compromessa (la revoca rende la history innocua: è l'opzione più semplice e sufficiente).
  4. Esporre Dokploy dietro HTTPS.

### 🔴 C-2 — Fallback silenziosi a password deboli nel compose

- **Dove:** `docker-compose.yml:11` e `:32-33` (`POSTGRES_PASSWORD=${POSTGRES_PASSWORD:-odoo}`, idem per user/db)
- **Requisiti violati:** riduzione errori di deploy; sicurezza; il fail-fast di `docker/entrypoint.sh:4-7` viene neutralizzato (la variabile risulta sempre valorizzata, anche se col default debole `odoo`).
- **Impatto:** se su Dokploy si dimentica `POSTGRES_PASSWORD`, l'istanza di produzione parte **senza errori** con password `odoo`. Peggio: se la variabile viene aggiunta in un secondo momento, Odoo non si connette più al DB già inizializzato con la password di default (PostgreSQL non ri-applica `POSTGRES_PASSWORD` a un data dir esistente) — scenario di lock-out con dati già in produzione.
- **Correzione:** rimuovere i default per tutti i segreti: `${POSTGRES_PASSWORD:?POSTGRES_PASSWORD must be set}` (sintassi supportata da Docker Compose), così il deploy fallisce subito e in modo esplicito. I default sono accettabili solo per valori non sensibili (`POSTGRES_VERSION`).

### 🔴 C-3 — Nessun versionamento delle release: tutto su tag mobili

- **Dove:** `.github/workflows/build-and-push-ghcr.yml:45` (unico tag `:latest`); `.env.example:9` (`ODOO_IMAGE=…:latest`); `Dockerfile:2` (`FROM odoo:18.0` senza digest)
- **Requisiti violati:** «versionamento controllato delle dipendenze», «rollback a una versione precedente», «tracciabilità e ripetibilità», «Non devono essere utilizzati riferimenti mobili non controllati».
- **Impatto:** un "Redeploy" su Dokploy prende qualunque cosa sia l'ultima immagine, senza corrispondenza certa con un commit; il rollback è impossibile (la `:latest` precedente è persa); `odoo:18.0` upstream cambia nel tempo, quindi due build dello stesso commit possono produrre immagini diverse.
- **Correzione:** nel workflow taggare ogni build anche con SHA e/o tag Git (es. `ghcr.io/…:sha-${{ github.sha }}` e `:vX.Y.Z` sui tag) usando `docker/metadata-action`; in produzione puntare `ODOO_IMAGE` a un tag immutabile e aggiornarlo come modifica versionata; pinnare la base con digest (`FROM odoo:18.0@sha256:…`) aggiornandola via PR (es. Renovate/Dependabot).

### 🔴 C-4 — La CI pubblica l'immagine di produzione a ogni push su `main`, senza alcun controllo

- **Dove:** `.github/workflows/build-and-push-ghcr.yml:3-13` (trigger: `push` su `main`/`master`); nessun altro workflow presente
- **Requisiti violati:** l'intera sezione «automazioni GitHub» (verifica codice, manifest, dipendenze, installazione, aggiornamento, test, avvio Odoo — sulle PR); «flusso di collaborazione» (punti 6-10); «ridurre gli errori di deploy».
- **Impatto:** qualunque push su `main` (incluso un merge frettoloso o un agente mal configurato) produce **immediatamente** la nuova immagine che il prossimo redeploy porterà in produzione, senza che nulla abbia verificato che Odoo si avvii o che i moduli si installino. Il flusso PR-based descritto nel brief non ha alcun supporto tecnico.
- **Correzione:** aggiungere un workflow di CI su `pull_request` che almeno: (a) lint (ruff + pylint-odoo), (b) validazione manifest/struttura moduli, (c) avvio di Odoo+Postgres in job e `odoo -i <moduli> --stop-after-init` con test (`--test-enable`), (d) build dell'immagine senza push. Attivare la branch protection su `main` (review obbligatoria + status check). Il push su GHCR resta solo sul merge in `main` (o sui tag di release, coerente con C-3).

### 🟠 C-5 — Moduli OCA copiati a mano nel repo, nessuno standard per i repository esterni

- **Dove:** `addons/mail_debrand/`, `addons/web_chatter_position/`, `addons/website_odoo_debranding/` (codice OCA vendorizzato; nessun riferimento a repo di origine, commit o versione oltre a quella nel `__manifest__.py`); nessun `repos.yml`/`.gitmodules`/script di fetch nel repo
- **Requisiti violati:** «i moduli OCA … non devono essere duplicati manualmente in ogni progetto»; «standard rigidi per i repository esterni»; «riferimenti fissati a commit, tag o release precise».
- **Impatto:** aggiornare un modulo OCA significa ricopiare file a mano (non tracciabile, non ripetibile, diff enormi); non c'è distinzione tra le tre categorie di addon richieste; ogni progetto derivato dal template divergerà a modo suo.
- **Correzione:** adottare lo standard proposto in §5 (git-aggregator con `repos.yml`, eseguito al build dell'immagine), rimuovere i moduli vendorizzati e dichiararli come dipendenze pinnate. Separare le directory: `addons/custom/` (cliente, nel repo), `addons/external/` (generata dal fetch, in `.gitignore`).

### 🟠 C-6 — Dipendenze Python non pinnate e installazione non deterministica

- **Dove:** `requirements.txt:1` (`pandas` senza versione); `Dockerfile:29-30` (`find /mnt/extra-addons -name 'requirements.txt' -exec pip install …` senza vincoli né lock)
- **Requisiti violati:** «versioni fissate o vincoli di versione controllati»; «ricostruire in momenti diversi un ambiente con le stesse versioni»; «controlli per individuare dipendenze mancanti, incompatibili».
- **Impatto:** ogni build può installare versioni diverse (es. un major bump di pandas) senza alcuna modifica al repo; i `requirements.txt` per-modulo vengono installati in ordine arbitrario e possono sovrascriversi a vicenda senza errore; nessuna verifica di compatibilità con le versioni vendorizzate da Odoo 18.
- **Correzione:** pinnare tutte le dipendenze (`pandas==2.2.*` o exact pin); valutare `pip-compile` (pip-tools) per generare un lock file unico che aggreghi root + moduli; installare con `--constraint` dai vincoli ufficiali Odoo 18 (`https://raw.githubusercontent.com/odoo/odoo/18.0/requirements.txt`) per evitare conflitti col core. Chiedersi inoltre se `pandas` serva davvero come baseline del template: è un pin pesante che ogni progetto erediterebbe.

### 🟠 C-7 — Routing Traefik fragile: rete non dichiarata e websocket senza TLS

- **Dove:** `docker-compose.yml:14-25`
- **Problemi:**
  1. Il servizio `odoo` è su **due reti** (`internal` + `dokploy-network`) ma manca la label `traefik.docker.network=dokploy-network`: Traefik può scegliere l'IP della rete `internal` (non raggiungibile da Traefik) e il routing fallisce, tipicamente in modo intermittente al riavvio dei container.
  2. Il router `${PROJECT_NAME}-ws` (`:20-22`) ha `entrypoints=websecure` ma **non** `tls.certresolver=letsencrypt` né `tls=true`: a seconda della configurazione dell'entrypoint di Dokploy il router websocket può risultare senza TLS e il canale `/websocket` (chat, bus, notifiche real-time) non funzionare.
  3. Il router `-ws` non è associato esplicitamente al service `-ws` (`traefik.http.routers.${PROJECT_NAME}-ws.service=${PROJECT_NAME}-ws` mancante): con due service definiti sullo stesso container, Traefik non può dedurre l'associazione e il routing verso la porta 8072 non è garantito.
- **Requisito violato:** «compatibilità con reverse proxy e HTTPS» (funzionante e ripetibile).
- **Correzione:** aggiungere le tre label mancanti. Verificare dopo il fix che `wss://<DOMAIN>/websocket` risponda.

### 🟠 C-8 — Nessun supporto al Coding Agent: né istruzioni né ambiente di test

- **Dove:** assenza di `AGENTS.md`/`CLAUDE.md`/`.github/copilot-instructions.md`; nessun compose/script di sviluppo o test; `addons/readme.md` vuoto
- **Requisiti violati:** le sezioni «supporto al Coding Agent», «ambiente temporaneo di sviluppo e test» e le responsabilità dell'agent.
- **Impatto:** l'agent non ha modo di sapere convenzioni, comandi di test, attività vietate, né di verificare che un modulo si installi prima di aprire la PR — esattamente ciò che il brief vuole evitare. È il gap che rende il template inutilizzabile per il flusso di lavoro descritto.
- **Correzione:** (a) creare un `AGENTS.md` come sorgente unica agent-agnostica (letto nativamente da Copilot coding agent, Codex e Antigravity), con `CLAUDE.md` ridotto a un rimando/include per Claude Code; contenuti minimi: architettura, versione/edizione Odoo, struttura addons, comandi per ambiente temporaneo e test, criteri di completamento, attività vietate (merge, deploy, segreti); (b) aggiungere `docker-compose.dev.yml` (o script `scripts/dev-env.sh`) che avvii Odoo+Postgres effimeri con bind-mount di `addons/` e un comando standard di test (`odoo -d test -i <modulo> --test-enable --stop-after-init`), riusando lo stesso Dockerfile e le stesse dipendenze della produzione come richiesto dal brief.

### 🟠 C-9 — Procedure di backup/ripristino incomplete e in parte scorrette

- **Dove:** `README.md:151-196` (§3.1, §3.3)
- **Problemi:**
  1. DB e filestore non sono trattati come **unità logica**: i due comandi sono indipendenti e nulla garantisce coerenza (il brief la richiede esplicitamente); il filestore viene copiato **a caldo** con `docker cp` mentre Odoo scrive.
  2. Nessuna pianificazione/automazione, nessuna conservazione esterna, nessuna verifica periodica del ripristino, nessuna procedura per deploy fallito — tutti punti richiesti dal brief.
  3. `pg_restore --clean` senza `--if-exists` fallisce su oggetti mancanti; i nomi container `<PROJECT_NAME>-db-1` sono un'assunzione che su Dokploy può non reggere (Dokploy può rinominare i servizi compose).
- **Correzione:** documentare (ed eventualmente scriptare in `scripts/backup.sh`) una procedura unica: stop del solo servizio Odoo (o modalità manutenzione) → `pg_dump` → archiviazione del volume filestore (`docker run --rm -v ${PROJECT_NAME}_odoo-data:… tar`) → riavvio; definire dove vengono conservati i backup (fuori dal nodo) e la procedura di rollback (immagine precedente + restore coerente), collegandola a C-3.

### 🟡 C-10 — `config/odoo.conf` incompleto per la produzione

- **Dove:** `config/odoo.conf`
- **Problemi:** manca `list_db = False` (con `admin_passwd` e `db_name` impostati, il database manager resta esposto su `/web/database/manager`: superficie di attacco inutile in produzione); manca `data_dir = /var/lib/odoo` esplicito (oggi funziona per default dell'immagine, ma il brief chiede che la persistenza sia dichiarata, non implicita); manca `dbfilter`; `workers = 2` è fisso e non parametrizzato rispetto alle risorse del nodo (il brief chiede risorse assegnate per istanza tra i parametri variabili).
- **Correzione:** aggiungere `list_db = False`, `data_dir` esplicito, e rendere `workers`/limiti parametrizzabili via env con default sicuri.

### 🟡 C-11 — Parametri dichiarati ma non effettivi

- **Dove:** `.env.example:20` e `README.md:58` dichiarano `ODOO_VERSION=18.0`, ma `Dockerfile:2` ha `FROM odoo:18.0` hardcoded: la variabile non ha alcun effetto.
- **Impatto:** contraddice «gli elementi variabili devono essere chiaramente identificati»: un utente del template che cambi `ODOO_VERSION` non ottiene nulla.
- **Correzione:** o usare `ARG ODOO_VERSION` nel Dockerfile (passata dal workflow), o rimuovere la variabile da `.env.example`/README. Vista la natura del template (Odoo 18 only), la seconda è più onesta.

### 🟡 C-12 — Nessun limite di risorse a livello container

- **Dove:** `docker-compose.yml` (nessuna sezione `deploy.resources` / `mem_limit`)
- **Impatto:** su un nodo Dokploy multi-istanza (scenario dichiarato nel README §1.2, `PROJECT_NAME` «univoco sul nodo»), un'istanza fuori controllo (o una query PostgreSQL degenerata) può saturare il nodo e abbattere le altre istanze. Il brief richiede «gestione delle risorse» e «risorse assegnate» tra i parametri.
- **Correzione:** aggiungere limiti parametrizzati via env (`mem_limit`, `cpus`) per `odoo` e `db`, coerenti con i limiti applicativi già presenti in `odoo.conf`.

### 🟡 C-13 — Documentazione operativa incompleta rispetto al brief

- **Dove:** `README.md`, `CHECKLIST.md`, `addons/readme.md` (vuoto)
- **Cosa manca:** procedura di sviluppo/test/review/release; procedura di aggiornamento moduli e dipendenze; procedura di rollback; responsabilità Agent vs sviluppatore; mappa dei segreti (quali servono, dove si configurano, quali sono riservati alla produzione); origine e versione dei moduli esterni; standard per i repository esterni; strategia di osservabilità oltre al debug rapido.
- **Correzione:** colmare in coordinamento con C-5/C-8 (molti contenuti nasceranno da quelle decisioni). `addons/readme.md` vuoto va compilato (categorie di addon e convenzioni) o rimosso.

### 🔵 C-14 — Rilievi minori

1. `.gitignore:5` ignora `api tocken.txt` (typo per "token"): indica che segreti locali vivono come file di testo nella working directory — pratica da abbandonare a favore di un password manager o di `.claude/settings.local.json` (vedi C-1).
2. `.dockerignore:5` esclude `*.md` globalmente: innocuo oggi, ma esclude anche i `readme/*.md` dei moduli OCA; se un modulo futuro leggesse file `.md` a runtime, si romperebbe in modo poco diagnosticabile.

## 5. Raccomandazione: standard per i repository esterni

Il brief impone di scegliere **un** meccanismo obbligatorio prima dell'implementazione. Confronto rispetto ai vincoli dichiarati (pin a commit/tag, ripetibilità, compatibilità Coding Agent + GitHub Actions + Dokploy, repo pubblici/privati, niente git all'avvio di Odoo):

| Criterio | Git Submodules | **git-aggregator (`repos.yml`)** | Script custom |
|---|---|---|---|
| Pin esatto a commit | ✅ nativo | ✅ (`commit:`/`refs`) | ✅ (da implementare) |
| Cherry-pick di sole cartelle/moduli o patch OCA | ❌ (si prende tutto il repo) | ✅ (merge di più ref, patch) | ⚠️ da implementare |
| Ergonomia per Coding Agent | ⚠️ scarsa (stato submodule fonte classica di errori nelle PR degli agent) | ✅ un file YAML dichiarativo, diff leggibili | ⚠️ dipende dalla qualità dello script |
| Aggiornamento via PR leggibile | ⚠️ (bump di SHA opachi) | ✅ (diff su YAML) | ⚠️ |
| Compatibilità GitHub Actions | ✅ (`checkout` con `submodules: true`) | ✅ (step pip + `gitaggregate`) | ✅ |
| Compatibilità Dokploy | ✅ se il fetch avviene in CI/build (non serve nulla a runtime) | ✅ idem | ✅ idem |
| Repo privati | ⚠️ deploy key/PAT per submodule | ⚠️ PAT come build secret (uguale) | ⚠️ idem |
| Manutenzione dello strumento | ✅ nativo Git | ✅ standard de-facto OCA, manutenuto | ❌ a tuo carico |

**Raccomandazione: git-aggregator con `repos.yml`**, eseguito **al build dell'immagine** (in CI/Dockerfile multi-stage), mai all'avvio di Odoo. Motivi: è lo standard de-facto dell'ecosistema OCA (l'ambito principale di questo template), il file `repos.yml` è esattamente il «meccanismo dichiarativo» chiesto dal brief, i diff di aggiornamento sono leggibili in review, e i coding agent lo gestiscono molto meglio dello stato submodule. Schema di destinazione:

```
addons/
├── custom/        # moduli specifici cliente (versionati in questo repo)
└── external/      # generata da gitaggregate al build — in .gitignore
repos.yml          # dichiarazione pinnata di OCA + moduli riutilizzabili
```

Per i **moduli proprietari riutilizzabili** (repo privati): stesso `repos.yml`, con PAT read-only passato come build secret in GitHub Actions (`secrets` → `docker/build-push-action` `secret`), mai presente nell'immagine finale né su Dokploy.
